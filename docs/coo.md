# Morn COO 主管系统

> COO (Chief Operating Officer) — 任务拆解、执行调度、学习进化

## 6 级决策树

COO 根据输入难度进行 6 级决策：

| 级别 | 判定条件 | 行为 |
|------|---------|------|
| L1 | 简单问答（<3秒） | 直接回答，不需要工具 |
| L2 | 需工具调用 | 路由到合适组件，自动调用 |
| L3 | 需拆解子任务 | 拆解为 DAG 子任务，顺序/并行调度 |
| L4 | 需多 Agent 协作 | 组建 Agent 团队，分配角色 |
| L5 | 需人工审批 | 暂停执行，等待 HITL 审批 |
| L6 | 超长任务（13h+） | 启动 LongTaskEngine，断点续传 |

### 6 步分步理解 (Step1-6)

| 步骤 | 行为 |
|------|------|
| Step1 | 输入理解：分析用户意图、提取关键信息 |
| Step2 | 难度评估：判断适合的决策级别 L1-L6 |
| Step3 | 资源规划：确定需要的工具/Agent/模型 |
| Step4 | 权重决策：基于任务复杂度/风险/成本加权评分 |
| Step5 | 拆分执行：按决策树分配子任务 |
| Step6 | 结果聚合：合并各子任务输出，生成最终响应 |

### 权重决策策略

| 维度 | 权重 | 说明 |
|------|------|------|
| 复杂度 | 0.3 | 子任务数量/依赖深度 |
| 风险 | 0.3 | 涉及敏感操作/数据 |
| 成本 | 0.2 | Token 消耗预估 |
| 紧急度 | 0.2 | 用户期望响应时间 |

### 自动升降级逻辑

- **升级条件**: 子任务执行失败 → 自动升级到上级 Agent 重新规划
- **降级条件**: 连续 3 次同类任务成功 → 自动降级到更低级别快速执行
- **VLM 真实校验**: 视觉模型验证 GUI 操作结果，失败则升级到人工

## COO 学习机制 (Learn from Feedback)

COO 通过两种方式持续进化：

### 决策规则学习
每次决策后记录：
1. 用户反馈（positive/negative）
2. 决策路径（输入→拆解→执行→结果）
3. 关键词提取（中文停用词过滤）
4. 规则持久化到 SQLite

### 实时模式切换

| 模式 | 行为 |
|------|------|
| active | 自动执行，无需确认 |
| safe | 关键操作需审批 (HITL) |
| auto | 根据上下文智能切换 |

## 三阶段 Agent (Triphase Agent)

```
[Plan]      ─→  [Implement]     ─→  [Review]
  │                │                  │
  ├ 分析输入       ├ 执行步骤         ├ 验证结果
  ├ 拆解步骤       ├ 调用工具         ├ 检查质量
  ├ 评估资源       ├ 处理中间结果     ├ 建议改进
  └ 输出计划       └ 输出执行日志     └ 输出审查报告
```

每个阶段可独立配置人格和模型。

## HITL 审批 (Human-in-the-Loop)

关键操作需要人工审批：

| 操作类型 | 审批要求 |
|---------|---------|
| 文件系统修改 | 需审批 |
| 系统配置变更 | 需审批 |
| 网络请求 | 需通知 |
| 普通对话 | 自由执行 |

审批流程：暂停 → 通知用户 → 等待 → 继续/中止/修改。

## 超长任务引擎 (LongTaskEngine)

支持 13 小时以上的长时间运行任务：

| 功能 | 描述 |
|------|------|
| 进度持久化 | save_progress / load_progress |
| 断点续传 | resume_from_checkpoint |
| 心跳检测 | heartbeat (定时确认活性) |
| 最终一致性 | 失败可重试，保证任务最终完成 |

## 主管-专家调度 (Orchestrator)

```
主管 Agent ─┬─ 专家 Agent 1 (数据分析)
            ├─ 专家 Agent 2 (代码生成)
            ├─ 专家 Agent 3 (报告撰写)
            └─ 专家 Agent N (...)
```

动态调度策略：
1. 主管接收任务 → 分析需求
2. 从专家池匹配最合适的 Agent
3. 派发子任务 → 收集结果
4. 合并输出 → 返回给用户

## 共识机制 (Consensus)

多 Agent 协作时通过共识文件接力达成一致：

| 轮次 | 行为 |
|------|------|
| 1 | 各 Agent 独立输出分析 |
| 2 | 读取其他 Agent 的输出，调整立场 |
| 3 | 合并共识，输出最终结果 |
| N | 循环直至达成多数一致 |

## Dual-LLM 安全管线

```
[Guard LLM] ─→ 输入安全检查 ─→ [Judge LLM] ─→ 输出安全审查 ─→ [Executor]
```

| 阶段 | 职责 |
|------|------|
| Guard | 输入安全检测：注入/越狱/敏感内容过滤 |
| Judge | 输出安全审查：合规/伦理/隐私校验 |
| Executor | 安全执行：仅通过双重审查的操作才实际执行 |
